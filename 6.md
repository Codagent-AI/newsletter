### Computational vs. Inferential Axis

> - SOURCE: https://martinfowler.com/articles/harness-engineering.html
> - EVIDENCE: Böckeler splits guides and sensors along a second axis:
>   - Computational: fast, deterministic; runs in milliseconds to seconds; reliable results.
>   - Inferential: slower, non-deterministic; slower and more expensive; results are more non-deterministic.
> - STRUCTURE: This is orthogonal to guides vs. sensors.
> - MATRIX:
>   - Computational guides: LSPs, CLIs, scripts, codemods, OpenRewrite.
>   - Computational sensors: eslint, semgrep, coverage, dep-cruiser, ArchUnit, mutation testing.
>   - Inferential guides: principles, CfRs, rules, reference docs, how-tos, skills.
>   - Inferential sensors: review agents, drift detection runners.
> - CLAIM: AI validation and AI code review mostly live in the inferential sensors quadrant.
> - DRAFT SLOT: Use this as framing for where Agent Validator and review agents fit.

## My Workflow

> PAUL:
>
> "Make it work, make it right, make it fast" - Kent Beck.
>
> As I wrote about before, it starts with SDD. I spent a *lot* of time on this, producing a detailed, thorough spec that includes an overview, multiple spec files, a design doc, and a task breakdown.
>
> When I go to implement it, I don't just say "implement this spec. Here is the full workflow:
>
> First, validate Code quality. Is the code readable and maintainable. No unnecessary duplication. This happens first, using /simplify skill skill. Also checks for performance issues. The agent session that implemented the code runs the reviews using subagents and then makes improvements.
>
> Then Agent Validator runs static checks and AI reviews in paralellel. Static checks: linter, type safety, build, tests, etc).
>
> AI reviews for the following:
>   1. Bugs. Check for logic errors, performance issues, and resource leaks.
>   2. Security issues and error handling gaps.
>   5. Task compliance.  Did the agent implement what was in the spec?
>
> These run in separate headless CLI sessions. The sessions *don't have to be* Claude. And in fact, since my evals pegged claude as a poor performing reviewer, I exclusively use GH Copilot or Codex. It's a nice way to avoid bumping up against that 5 hour Claude limit.
>
> Why three agents? Each one gives a more focused and comprehensive review (higher recall). My evals showed that running the bug check separately while combining security and error handling leads to optional quality vs cost.
>
> And then there's the "assumptions review". During implementation of each task, I have the agent note down assumpions it made - possible shortcuts, anything that wasn't fully clear in the spec. And then the *original* (and likely smarter) planner agent reviews all the assumptions and makes corrections where needed - surfacing anything *it's* not sure about at the end back up to me.
>
> But the validations don't stop there. These are from specialized products dedicated to code review. Advanced fetures. They will catch things that even local revivews with Agent Validator can't. And my workflow waits for these and then addresses these issues too.
>
> That must take me a long time to get through this whole workflow, right? It takes me no time, it happens while I'm doing something else (like planning the next change). All the steps I mentioned above of these run automatically and *autonomously* after the agent implements the task, in sequence, thanks to Agent Runner. Even the PR creation, and then waiting for CI and Agent Reviews.
>
> None of the steps get skipped. Ever. (Even if the agent is feeling lazy).
>
> So when I come back to the task, I can trust that it's already been verified for correctness and quality. I can instead focus on what I do best, steering the ship (link to https://codagent.beehiiv.com/p/the-middle-agentic-path).
