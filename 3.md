# Short Leash, Long Game

## The right scope isn't a constraint. It's the architecture.

The pitch for long-running agents is seductive: hand off the whole project, check back when it's done. Some teams are doing exactly this — autonomous agents working for hours or days, pulling context from codebases, writing code, running tests, opening PRs. The demos are impressive.

But there's a failure mode that doesn't show up in demos. It shows up on day three, when you dig into why something feels slightly off, and discover the agent made a plausible-sounding assumption on day one — about a data model, an API contract, an interpretation of a requirement — and built everything downstream from it. The assumption was wrong. The code looks clean. Tests pass. The output is authoritative and well-structured and confidently incorrect. Glen Rhodes calls it "silent drift." The Medium piece calls it "wrong early assumptions that compound invisibly." I call it building with Ralph Wiggum.

The problem isn't the model. It's the scope. At project scale, the gap between what you specified and what the agent inferred is enormous. Every inference is a potential error, and every error propagates forward through everything built after it. By the time you review, unwinding is expensive. The solution isn't a better prompt or a longer context window. It's a shorter leash — not because agents are bad, but because tight scope is what makes tight verification possible. And tight verification is the only real defense against silent drift.

## Analysis: Feature at a Time

The right response to silent drift isn't more supervision of agent output. It's scope discipline before the agent starts.

I keep coming back to this when I watch teams hit the same wall. They're not using the wrong model. They're not writing bad prompts. They're giving agents the wrong unit of work. Everything else — verification strategy, context management, human review — depends on getting that right first. Get scope wrong and nothing else compensates.

Here's why it matters mechanically. An agent working on a feature with a clear spec and a defined "done" has two properties that make verification tractable: bounded context and a concrete exit condition. You can write tests against it. You can run a validator that checks the output against the original spec. You can read the result yourself and ask "did this do the thing I asked?" without holding the entire codebase in your head. If something drifts, you find out fast and the cost to fix it is low — usually a corrected spec and a re-run.

An agent working on a project has neither property. You can't fully specify a whole project before you've built any of it — the spec is necessarily incomplete when the work starts. The exit condition is weeks away. Every assumption the agent makes in week one becomes a constraint that shapes week two. By the time you review, you're not evaluating isolated decisions. You're evaluating an architecture of compounded inferences, where the first wrong assumption is buried under everything that followed from it. The cost of being wrong is now the cost of unwinding all of that. Most teams don't unwind. They ship it.

The failure mode isn't a crash. It's worse than a crash. Rhodes puts his finger on exactly why: LLM outputs look authoritative. Clean, well-structured, logically coherent. No stack traces, no error flags, no obvious signs that something went wrong. The agent ran for three days and produced a professional-looking result. The assumptions it made are invisible unless you specifically go looking for them — and by then you've been living inside this system long enough that questioning the foundations feels like questioning your own judgment. Silent drift isn't just technically dangerous. It's psychologically dangerous, because the output is calibrated to look like it should be trusted.

---

So what's the right unit? "Feature" needs precision here, because not all features are the same size, and product roadmap features can be enormous.

The test I use: if you handed the spec to a capable engineer who'd never seen your codebase, could they build it without a single clarifying question? If yes, it's probably the right size. If they'd have a list before they started — what's the schema, which component library, how does auth work here — then your spec is a project brief with a feature-shaped name. And every one of those unanswered questions is an assumption the agent will fill in for itself, invisibly, in whatever way seems most plausible given its training.

"Add webhook notifications with retry logic and exponential backoff, firing on these three event types, storing delivery status in this table" is a feature. "Build the notifications subsystem" is not. The difference isn't just size — it's specifiability. One can be fully defined before work starts. The other can't, and any attempt to pre-define it will be wrong in ways you won't discover until you're deep in.

This isn't artificial constraint. It's the granularity at which feedback loops work. A feature is completable and evaluable in a single session. You run it, you check it, you know whether it worked. That tight cycle is what makes agentic development fast in practice — not because each feature goes faster than writing by hand (though it often does), but because when something goes wrong, you catch it before it has compounded. Speed in agentic development isn't about reducing the time per cycle. It's about running more cycles, each one recoverable.

---

Once you've scoped the work correctly, a second question: when the feature is done, what's the human's job?

Most teams answer: code review. Read the diffs, check the logic, catch the bugs. I think this is the wrong answer — not because agents write perfect code, but because humans are structurally bad at code review, and always have been.

Developers miss 30-40% of defects in code review under normal conditions. Under time pressure, that climbs. Reviews of large diffs degrade quickly — tracking state across hundreds of lines of unfamiliar code exceeds what working memory can handle, and we compensate by reviewing what we can evaluate quickly: naming, style, obvious logic errors. The bugs that survive code review are usually the subtle ones, the ones that require holding the full context simultaneously. Which is precisely the class of bug that agents are most likely to introduce.

There's also an authority bias problem that's especially acute with AI output. Agent-generated code looks like it was written by someone who knew what they were doing. Consistent style, well-commented, follows your existing patterns. It signals competence. And signals of competence cause reviewers to look less carefully — if you think the author is good, you check that the overall approach makes sense and trust the details. With AI output, that heuristic is actively misleading. The agent produces equally competent-looking code whether the implementation is correct or whether it's been silently wrong about a requirement since the first inference.

A validator and a test suite catch correctness issues more reliably than human eyes. Not because they're smarter, but because they're systematic in a way humans can't be. A type-checker catches every type error, not just the ones a reviewer happened to notice. A linter flags every rule violation, not just the ones that stood out on a tired Friday afternoon. Custom checks verify every constraint in the spec, not just the ones that seemed worth double-checking. The thing validators can't do is judge intent — whether the code is solving the right problem. That's what humans are actually good at, and it's a completely different job.

---

The job I'm describing is product judgment: is this the right feature to build next? Did this solve the real problem? Is the direction we've been going still the right direction, or did building this feature reveal something that should change what comes next?

These questions require understanding the goal, not just the output. They require holding the product's direction in mind across multiple features, noticing when the accumulated work is pulling somewhere unexpected, and having the judgment to redirect before too much is built in the wrong direction. No validator can answer them. No test suite flags when the product is technically correct but strategically wrong. That's judgment, and it should not be spent on diff review.

The right place for that judgment is at the boundary between features — after one thing is done and before the next starts. That's when the relevant questions are live. Did the feature we just built change what the next one should be? Did we learn something that should update the spec for the next cycle? Is this still the highest-value thing to build? The boundary is the decision point, and the human should own it completely.

This is what the Medium article is really getting at when it talks about redistributing judgment earlier. But the framing is slightly off. The judgment doesn't just move earlier in time — it moves to a different kind of question. It moves from "is this code correct" (which humans do poorly and tools do well) to "is this the right thing to build" (which tools can't answer and humans are uniquely suited for). It's not a temporal redistribution. It's a reassignment — to work that actually needs human judgment, and away from work that only feels like it does.

---

Here's the implication that I don't think gets enough attention: direction review at the feature boundary doesn't just set the next feature. It makes the next feature better.

When you ask "what did we learn from this?" before speccing the next cycle, you're surfacing the assumptions the agent made that you didn't specify. Some of those were correct. Some were wrong. Some were right for this feature but reveal constraints you hadn't thought to articulate. When you capture those and fold them into the next spec, you've just made the next execution more accurate — without any improvement in the model.

Each boundary makes the next spec more precise. Each spec makes the next execution more accurate. Each execution generates better information for the next direction review. It compounds. This is the opposite dynamic from project-scale work, where each unchecked inference potentially compounds errors. Feature-by-feature compounds improvements. If you're disciplined about it, your specs get sharper over weeks, your direction-setting gets more precise, and the thing you're building gets progressively more aligned with what you actually want — because you're correcting alignment every cycle instead of discovering misalignment after months.

---

The MetaGPT researchers tested this directly. When they added structured planning artifacts to the front of each execution cycle — before any code was written — executability jumped from 1.0 to 4.0 out of 4, and required human revisions fell from 10 to 2.5. Same model. Same codebase. The only variable was the planning work that happened at the boundary before execution started.

That's where the leverage is. Not in a better model, not in more code review, not in a longer context window. In the work you do before you start. The agent will execute faithfully against whatever you give it. The question is always: are you giving it the right thing?

Writing a sharp feature spec — one that contains the agent's assumptions, has a verifiable exit condition, and captures the constraints that actually matter — is a skill. It's not the same skill as writing code. It's closer to thinking clearly about what a thing is for before you build it. Most developers never had to develop it, because iterative coding environments rewarded adjustment over preparation. Agents don't remove the need for iteration. But they shift the leverage point decisively toward preparation. A developer who writes a sharp spec spends most of their effort on the problem. A developer who writes a vague spec spends most of their effort managing the consequences of that vagueness. The gap has nothing to do with which model you're running.

## My Take

You don't need a long-running agent. You don't need Ralph Wiggum.

Ralph Wiggum, for the uninitiated, is the kid from The Simpsons who does things with complete sincerity and total disconnection from what's actually needed. He's helping in the sense that he's doing *something*. It's just rarely the right thing. A long-running autonomous agent — given a vague project brief, left to run unsupervised, surfacing output after days of execution — is Ralph Wiggum. It produces things. The things look like code. They might even pass your tests. They're just not always solving the problem you actually have.

Long-running agents are a solution to the wrong problem. The problem they solve is "I don't want to stay in the loop." But staying in the loop — at the right points, at feature boundaries — is what keeps work from drifting. An agent that runs for three days without human input isn't more autonomous. It's just less checkable. The right model isn't "go build me a product and I'll review it when you're done." It's "go build this one thing, then we'll decide what comes next." Scope it right, verify it, set direction for the next one. That loop, done well, is faster and more reliable than any long-running autonomous agent — because every iteration is evaluable, every error is recoverable, and the human's judgment lands where it actually matters.
