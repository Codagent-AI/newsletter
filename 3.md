# The Middle Agentic Path

## Both sides are wrong about human-in-the-loop

There are two loud camps in agentic coding right now.

Camp one says you need a human reviewing every line of AI-generated code before it ships. This is the engineering establishment position — responsible, mature, safe. It's also increasingly untenable. AI-assisted teams are generating 98% more pull requests while reviewer capacity stays flat. The queue is breaking.

Camp two says the answer is more autonomy. In February 2026, every major platform shipped multi-agent capabilities within the same two-week window: Claude Code launched Agent Teams, Cursor shipped Background Agents, GitHub Copilot added `/fleet`, OpenAI Codex got subagents, Factory AI built a "droid army." The pattern is the same everywhere: an inner agentic loop per agent — implement, validate, fix, repeat — with an outer orchestrator distributing tasks across parallel agents, each isolated in its own git worktree. Steve Yegge's Gas Town pushes this further: 20-30 simultaneous agents with specialized roles, and a vision called "The Wasteland" that federates thousands of these teams together. The demos are impressive. But do you really need all this?

I think there's a third position: the human should be in the loop for every change — but not reviewing code line by line. Writing sharp specs, exercising taste, maintaining architectural coherence, and deciding what to build next based on what you just learned. The human's job isn't to read diffs. It's to steer.

## Code review is dead

I [argued in December](https://medium.com/@paul.caplan/the-era-of-mandatory-code-reviews-is-over-5c5593ba5613) that the era of mandatory code reviews was over, and [others](https://thenewstack.io/traditional-code-review-is-dead-what-comes-next/) argued [more recently](https://addyo.substack.com/p/code-review-in-the-age-of-ai) along similar lines. The case is getting stronger every day.

The math is simple. DORA's 2025 report found that AI-assisted teams produce 98% more PRs while review time increases 91%. PR sizes are up 154%. Armin Ronacher — the creator of Flask — [called review "the final bottleneck"](https://lucumr.pocoo.org/2026/2/13/the-final-bottleneck/) and cited OpenClaw's 2,500+ open PRs as evidence the queue is breaking. Using queuing theory: when arrival rate exceeds service rate, the queue grows without bound. That's where we are.

But the deeper problem isn't throughput. It's that most of what code review is *for* can now be done better by machines.

Think about what a reviewer actually checks. Maintainability — complexity, file size, dead code — is largely enforceable with static analysis and linter rules. Adherence to team and project standards is increasingly handled by AGENTS.md files and rule configurations; AI reviewers cross-reference these too, resulting in very high compliance. Catching bugs, security vulnerabilities, swallowed errors — this is exactly where AI code review tools excel, systematically and consistently. The parts of code review that are mechanical have been automated. What's left is judgment — and humans have never been particularly good at exercising it through code review.

Developers miss 30-40% of defects in code review under normal conditions. Under time pressure, that climbs. Reviews of large diffs degrade fast — tracking state across hundreds of lines of unfamiliar code exceeds working memory, and we compensate by checking what we can evaluate quickly: naming, style, obvious logic errors. The subtle bugs survive. Which is precisely the class of bug that AI agents are most likely to introduce.

There's also an authority bias problem that's especially acute with AI output. Agent-generated code looks like it was written by someone who knew what they were doing. Consistent style, well-commented, follows existing patterns. It signals competence. And signals of competence cause reviewers to look less carefully — if you think the author is good, you trust the details. With AI output, that heuristic is actively misleading. The agent produces equally competent-looking code whether the implementation is correct or whether it's been silently wrong about a requirement since the first inference.

A type-checker catches every type error. A linter flags every rule violation. A security scanner checks every dependency. A human catches the ones they happen to notice on a Friday afternoon. For correctness verification, deterministic tools and AI reviewers are systematically better than human eyes — not because they're smarter, but because they don't get tired, don't rubber-stamp, and don't have an afternoon slump.

The tooling is already here. [One in five code reviews on GitHub now uses Copilot](https://github.blog/ai-and-ml/github-copilot/60-million-copilot-code-reviews-and-counting/) — 60 million reviews since launch. Greptile [reports an 82% bug catch rate](https://www.greptile.com/benchmarks) in benchmarks across real-world PRs, hitting 100% on high-severity bugs. The Linux Foundation told CodeRabbit it catches [90% or more of customer-facing bugs](https://www.coderabbit.ai/case-studies/how-coderabbit-helped-the-linux-foundation-streamline-code-reviews-and-accelerate-development) and cut their review time in half. On [Martian's independent benchmark](https://codereview.withmartian.com/) — the only one not run by a vendor — Qodo scored 64.3%, ten points ahead of the next competitor. These numbers aren't perfect, and every vendor's own benchmark conveniently ranks themselves first. But the trajectory is clear. AI code review tools are already catching categories of bugs that human reviewers routinely miss, at a pace and consistency that humans can't match.

To be clear: I'm not saying human review has zero value. A senior engineer reviewing a tricky concurrency change or a security-critical auth flow — that's time well spent. What I am saying is that making human review a blocking gate on every PR does more harm than good. It's a tradeoff, and right now the costs — the five-day queues, the rubber-stamping, the false sense of security from reviews that miss 30-40% of defects — outweigh the benefits for the majority of changes shipping today.

So if not mandatory human review on every change, what? I'll come back to that.

## You don't need a colony or a swarm

On the other end of the spectrum, some want to remove the human from the loop entirely.

Gas Town is the most ambitious version of this vision. Yegge's system orchestrates 20-30 parallel Claude Code instances with specialized roles — a Mayor as the primary interface, Polecats as ephemeral workers, a Refinery managing the merge queue so parallel work doesn't collide. Work is tracked through "Beads," lightweight JSON items stored in Git. The Wasteland extends this further: a federated network linking thousands of Gas Towns, with a shared work board and a reputation system. His thesis, borrowed from biology: "When work needs to be done, nature prefers colonies."

Maybe. But nature also spent millions of years evolving the coordination mechanisms that make colonies work. We've had coding agents for about eighteen months.

The failure mode here isn't hypothetical. Glen Rhodes [describes it precisely](https://glenrhodes.com/agent-orchestration-failure-modes-silent-drift-reconciliation-and-the-supervision-mindset-shift/): in multi-agent systems, Agent A's flawed output becomes Agent B's unvalidated input, and LLM outputs look authoritative — clean, well-structured, logically coherent — even when carrying compounded errors. He calls it "silent drift." The more agents you run in parallel, the more surfaces for drift, and the less visibility you have into any single agent's assumptions.

A [recent piece on long-running agents](https://medium.com/ai-the-long-run/what-long-running-agents-change-and-what-they-dont-0788ff58d1e1) makes the complementary point: vague specs that cause minimal friction in synchronous work become catastrophic in async execution. Wrong early assumptions compound throughout the run, produce output that passes tests, and solve the wrong problem. The failure mode is invisible until it's expensive.

I've seen this movie before. In 2015, microservices were the answer to everything. Every startup needed Kubernetes, service meshes, distributed tracing. The architecture was genuinely good — for Google, for Netflix, for companies operating at massive scale with hundreds of engineers. For a five-person startup, it was a catastrophe of accidental complexity. The infrastructure to run the system consumed more engineering time than the product itself.

There are cases where this kind of swarm makes sense. Porting a large codebase to a new language or framework, where you know 100% of the requirements because the existing system *is* the spec — that's a genuine use case for massive parallelism. The requirements are fully defined, the exit condition is "it behaves identically," and there's little room for silent drift because you're not making design decisions, you're translating. Large-scale refactoring and migration work fits the same pattern.

But for the vast majority of developers building products today — where requirements are discovered, not given — running 30 parallel agents is likely solving a problem you don't have with complexity you can't afford.

There's a Simpsons character called Ralph Wiggum who does things with complete sincerity and total disconnection from what's actually needed. He's helping in the sense that he's doing *something*. It's just rarely the right thing. A long-running autonomous agent — given a vague project brief, left to run unsupervised — is Ralph Wiggum. A team of thirty Ralph Wiggums is not an improvement. It's thirty things being done with complete sincerity and total disconnection from what you actually need, all at once, at scale. And slapping mandatory code review on top doesn't fix it — you're just asking a tired human to evaluate whether thirty confidently-wrong things are correct by reading diffs on a Friday afternoon.

The question I think more people should be asking isn't "how many agents can I run?" It's "what is the human's actual job?"

## What the human actually does

Here's where I think both camps make the same mistake: they assume the human's primary role is either to review code or to get out of the way. The actual job is richer than either camp acknowledges — and almost none of it requires reading diffs line by line.

Start with what happens before the agent touches anything. The human writes the spec. Not a formality, not a box to check — the spec is the human's primary output. Osmani [calls it](https://addyosmani.com/blog/next-two-years/) "the leverage": "Ambiguous requirements propagate through dozens of parallel runs, each going slightly wrong in a slightly different direction." A sharp spec — one that captures the behavioral contract, the constraints that actually matter, and a testable exit condition — is the difference between an agent that builds the right thing and one that builds a plausible-looking wrong thing. This is a skill, closer to thinking clearly than to writing documents. Most developers never had to develop it, because iterative coding environments rewarded adjustment over preparation. Agents shift the leverage point decisively toward preparation.

The test I use for whether a spec is sharp enough: if you handed it to a capable engineer who'd never seen your codebase, could they build it without a single clarifying question? If yes, it's probably right. If they'd have a list of questions before they started — what happens when the webhook delivery fails three times, should the empty state show a placeholder or a CTA, does this need to work offline — then your spec is a project brief with a feature-shaped name. And every one of those unanswered questions is an assumption the agent will fill in for itself, invisibly, in whatever way seems most plausible given its training data.

Knowing what to specify means knowing what *not* to build. Kent Beck [puts it directly](https://tidyfirst.substack.com/p/programming-deflation): "When anyone can build anything, knowing what's worth building becomes the skill." Restraint is harder than execution now. Every feature you don't build is complexity you don't maintain, surface area you don't defend, context that doesn't crowd the agent's window. This is judgment that lives upstream of any code, and no amount of review downstream compensates for getting it wrong.

Then there's organizational memory — the context that lives in people's heads, not in the codebase. Which technical debt is tolerated for business reasons. Why the API is shaped the way it is. What the team tried last quarter and why it didn't work. As [Birgitta Böckeler writes](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html) on Martin Fowler's site, agents have "no awareness of what the team is trying to achieve, no intuition that 'we don't do it that way here,' and no organisational memory." That context doesn't belong in a code review comment. It belongs in the spec, in the constraints, in the agent's instructions — placed there by someone who carries it.

None of that requires looking at a single line of code. But the human's job doesn't end when the agent starts.

---

When a feature comes back, the human does look at it — but not the way a traditional code reviewer does. You're not tracing logic paths or checking edge cases. You're exercising taste. A thirty-second scroll through a PR tells you whether the solution is over-engineered, whether the abstractions feel right, whether this is code you'd want to maintain in six months. Beck [describes the feeling precisely](https://tidyfirst.substack.com/p/augmented-coding-beyond-the-vibes): "I feel good about the correctness & performance, not so good about the code quality." That's taste — the ability to sense that something is technically valid but contextually wrong. Agents produce code with "no aesthetic disgust at a 300-line function," as Böckeler puts it. Someone has to have that disgust.

You're also the architectural gatekeeper. Not reviewing implementation details, but asking whether this feature's approach is coherent with the system's direction. Does this create a dependency we'll regret? Does this abstraction make the next three features easier or harder? OpenAI's harness engineering team had one engineer [whose role was exactly this](https://openai.com/index/harness-engineering/) — maintaining architectural vision while "shipping code he doesn't read line by line."

And you're the bullshit detector. The person who looks at clean, passing, well-structured code and senses that something is off. Maybe the agent solved the wrong problem. Maybe it made an assumption that's invisible in the code but obvious if you know the domain. This isn't a skill you can automate, because it requires knowing things that aren't written down anywhere.

---

Here's the critical point: no spec is perfect. No matter how sharp your specification is, there will be gaps — edge cases you didn't think of, design decisions you left implicit, constraints you carry in your head but didn't write down. The agent will fill every one of those gaps with its best guess, silently, confidently. AI review tools will catch some of those — the ones that manifest as code-level issues. But the assumptions that are contextually wrong, aesthetically off, or architecturally misguided? Those pass every automated check. They look clean. They work.

This is why you need a human checkpoint after each change, before continuing. Not to review the code line by line — but to exercise judgment that no tool can automate. At that boundary, you're doing up to five things at once: exercising taste over whether the solution feels right, checking architectural coherence with the system's direction, detecting bullshit that passes every test but smells wrong, learning something from this cycle that sharpens the next spec, and deciding whether the next thing you planned is still the right next thing to build. None of these require reading every diff. All of them require being close enough to the work to have an informed opinion.

And this is exactly what breaks at colony scale. You can't exercise taste over thirty parallel agents' output — the volume overwhelms the intuition. You can't maintain architectural coherence when thirty streams are making independent micro-decisions. You can't smell that something is off when you're distant from all of it. And you can't learn from the last feature to sharpen the next spec when everything is running at once. The colony model doesn't just skip the human checkpoint — it makes it structurally impossible.

The sequential checkpoint is also what makes the work compound in the right direction. Each boundary review surfaces assumptions the agent made that you didn't specify. Some were correct. Some were wrong. Some reveal constraints you hadn't articulated. When you capture those and fold them into the next spec, you've made the next execution more accurate — without any improvement in the model. Feature-by-feature compounds improvements. Project-scale work compounds errors. The dynamics are opposite.

The MetaGPT researchers tested this directly. When they added structured planning artifacts at the front of each execution cycle — before any code was written — executability jumped from 1.0 to 4.0 out of 4, and required human revisions fell from 10 to 2.5. Same model. Same codebase. The only variable was the planning work that happened at the boundary before execution started. That's where the leverage is.

## Codagent Update

So if the human's job isn't reviewing code, something still has to. That's what Agent Validator is for.

Agent Validator is the verification engine I've been building as part of Codagent. It runs a hybrid validation pipeline — deterministic checks (your tests, linters, type-checkers, security scanners) combined with cross-agent AI review. The idea is simple: one AI model reviews code written by another. Claude implements a feature; Codex reviews it. Different model, different training data, different blind spots. In evaluation, cross-agent review hit F1 scores of 0.69-0.71 — substantially better than self-review, with significantly better token efficiency than throwing a single model at the problem with more passes.

The key distinction from PR-based AI review tools like CodeRabbit or Copilot Code Review: Agent Validator runs *during* development, not after a PR is opened. The agent writes code, Agent Validator runs checks and cross-agent reviews, the agent reads failures and iterates — in a tight loop until everything passes. No PR queue, no human bottleneck, no five-day wait. By the time you see the result, it's already been through a more consistent review process than most human reviewers can sustain.

This maps directly to the argument above. The human's job is direction-setting at the feature boundary. Agent Validator handles the correctness verification that humans were never great at anyway — and it does it during development, not as a gate after the fact. It's the thing that replaces mandatory code review. Not with "trust the agent," but with deterministic, repeatable, multi-perspective automated verification.

As I argued above, you need a good AI code review tool. The commercial options — CodeRabbit, Greptile, Qodo — are effective but not free, typically running ~$30/seat/month. Agent Validator gives you cross-agent review for cheap or free, if you already have a spare subscription to Codex or Copilot. It's open-source and works with whatever CLI tools you already have. Check out [codagent.dev](https://codagent.dev) to get started.
